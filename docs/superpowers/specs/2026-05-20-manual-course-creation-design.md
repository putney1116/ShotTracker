# Manual Course Creation

Add an in-app wizard to create a golf course from scratch when it isn't available via OpenStreetMap search. The wizard captures everything required for full distance-to-green features: per-hole par and front/center/back green GPS coordinates, plus tee rating/slope.

## Motivation

The current course flow depends on OpenStreetMap data via Overpass. Many real courses (e.g., Townsend Ridge Golf Club in Townsend, MA) are not mapped in OSM at all — Overpass returns zero results. Users have no way to add such courses today. This feature closes that gap while preserving the app's existing distance-to-green features by requiring the same GPS data that the OSM importer produces.

## User Flow

1. From CourseManagement, tap a new "Create Course Manually" button (placed under the existing "Search & Import Courses" button).
2. A linear wizard guides the user through six steps. The user can move forward only when the current step is valid.
3. On save, the new course appears in CourseManagement and is immediately usable for rounds — including distance-to-green.

**Wizard steps:**

1. **Name** — course name (required), city and state (optional).
2. **Hole count** — toggle 9 or 18.
3. **Pars** — list of holes with +/- buttons (default par 4, range 3..6).
4. **Greens** — for each hole sequentially, tap a satellite map to place front, center, and back pins. All three are required to advance.
5. **Tees** — at least one tee with name; rating (60..80) and slope (55..155) optional but validated if entered.
6. **Review** — summary; tap Save.

Hardware back / top-bar back on any step decrements one position (within Greens, between holes; otherwise between steps). Backing out from step 1 prompts a "Discard course?" confirmation dialog. Post-save: pop back to CourseManagement.

## Architecture

Single Compose screen with internal step state. One ViewModel owns the entire wizard. No persistence until the final save.

### Navigation

### No entity / DAO / domain-model changes

The existing schema already supports everything this wizard captures:

- `CourseEntity` (name, city, state)
- `HoleInfoEntity` (par, nullable front/center/back lat/lng)
- `TeeEntity` (name, color, rating, slope)

The wizard assembles in-memory drafts and calls a new repository method on save.

### Repository addition

Two new methods, one per layer:

```kotlin
// CourseRepository (interface) — takes domain models
suspend fun createCourse(course: Course, holes: List<HoleInfo>, tees: List<Tee>): Long
```

```kotlin
// CourseDao — takes entities, wrapped in @Transaction so all three inserts are atomic.
// Holes and tees come in with courseId = 0; the DAO method stamps the real ID after
// inserting the course, since the autogenerated key isn't known until that insert returns.
@Transaction
suspend fun createCourseWithHolesAndTees(
    course: CourseEntity,
    holes: List<HoleInfoEntity>,
    tees: List<TeeEntity>,
): Long
```

`CourseRepositoryImpl.createCourse` maps the domain models to entities (leaving `courseId = 0` on holes and tees) and calls the DAO's `@Transaction` method. The DAO inserts the course, gets the autogenerated ID, copies it onto each hole and tee via `.copy(courseId = newId)`, then inserts both lists. If any insert throws, Room rolls back the entire transaction so no orphan rows remain.

## Wizard State Machine

```kotlin
data class CreateCourseUiState(
    val step: WizardStep = WizardStep.Name,
    val name: String = "",
    val city: String = "",
    val state: String = "",
    val nameError: String? = null,        // e.g., "blank" or "duplicate"
    val holeCount: Int = 18,
    val pars: List<Int> = List(18) { 4 },              // index 0 = hole 1
    val greens: List<GreenDraft> = List(18) { GreenDraft() },
    val tees: List<TeeDraft> = emptyList(),
    val teeFormState: TeeFormState? = null,            // inline add/edit dialog
    val currentGreenHole: Int = 1,                     // 1-based, only used on Greens step
    val selectedGreenTarget: GreenTarget = GreenTarget.Front,
    val showDiscardDialog: Boolean = false,
    val isSaving: Boolean = false,
    val saveError: String? = null,
)

sealed class WizardStep {
    data object Name : WizardStep()
    data object HoleCount : WizardStep()
    data object Pars : WizardStep()
    data object Greens : WizardStep()       // iterates via currentGreenHole
    data object Tees : WizardStep()
    data object Review : WizardStep()
}

enum class GreenTarget { Front, Center, Back }

data class GreenDraft(
    val frontLat: Double? = null, val frontLng: Double? = null,
    val centerLat: Double? = null, val centerLng: Double? = null,
    val backLat: Double? = null,  val backLng: Double? = null,
) {
    val isComplete: Boolean
        get() = frontLat != null && centerLat != null && backLat != null
}

data class TeeDraft(
    val name: String,
    val color: String?,
    val rating: Double?,
    val slope: Int?,
)
```

### Transitions

- `onNext()` — runs per-step validation; if valid, advances. On Greens, increments `currentGreenHole` until `holeCount`, then advances `step` to `Tees`.
- `onBack()` — reverse; on Greens with `currentGreenHole > 1` decrements the hole instead of leaving the step. On Name, sets `showDiscardDialog = true`.
- `placeGreen(target, latLng)` — updates `greens[currentGreenHole - 1]` for the given target via `copy`, then auto-advances `selectedGreenTarget` to the next unfilled slot (Front → Center → Back; stays on Back if all placed).
- `selectTarget(target)` — manual override; tapping a target chip whose pin is already placed clears that pin (acts as undo).
- `onSave()` — only callable from `Review`. Builds domain objects, sets `isSaving = true`, calls `repo.createCourse(...)`. On success, emits a one-shot `CreateCourseEvent.Saved` event the screen collects; on failure sets `saveError` and resets `isSaving`.

### Per-step validation (gates `onNext`)

| Step       | Rule                                                                            |
|------------|---------------------------------------------------------------------------------|
| Name       | Trimmed length 1..80; no duplicate (single `getCourseByName` check).            |
| HoleCount  | 9 or 18 (toggle cannot produce an invalid value).                               |
| Pars       | Each par in 3..6 (already constrained by +/- buttons reused from CourseManagement). |
| Greens     | `greens[currentGreenHole - 1].isComplete == true`.                              |
| Tees       | At least one tee with non-blank name. Rating, if entered, parses to a Double in 60..80; slope, if entered, parses to an Int in 55..155. |
| Review     | Always valid.                                                                   |

When `setHoleCount(n)` is called, both `pars` and `greens` lists are resized to `n` entries (truncate or pad with defaults).

### State preservation

The ViewModel is `@HiltViewModel`-scoped to the nav entry. **Process death mid-wizard is not preserved in v1** — the user lands back on CourseManagement with no draft on relaunch. This is an accepted limitation; a long wizard plus three Google Maps tap surfaces makes SavedStateHandle persistence complex, and the use case (creating a course in a single sitting) makes loss tolerable.

## Map Interaction (Greens Step)

Full-screen `GoogleMap` (satellite) with a fixed top bar and a fixed bottom bar overlaying the map.

**Top bar**: `"Hole {N} of {holeCount} — Place {Front|Center|Back}"`

**Bottom bar**: three target chips (Front / Center / Back) showing placed/unplaced state, plus Prev/Next hole buttons. Next is disabled until all three pins are placed for the current hole.

**Tap handling**:
- `GoogleMap.onMapClick = { latLng -> vm.placeGreen(selectedGreenTarget, latLng) }`
- After each tap, `selectedGreenTarget` auto-advances to the next unfilled target on the current hole (Front → Center → Back). When all three are placed, `selectedGreenTarget` remains on Back so further taps re-place Back.
- Tapping an already-placed target's chip clears that pin and sets `selectedGreenTarget` to it.
- When `currentGreenHole` changes (Next Hole / Prev Hole), `selectedGreenTarget` resets to the first unfilled target on the new hole (Front for an untouched hole; whichever is still missing if the hole was partially completed before).

**Rendering**: three `Marker`s with the same color scheme used in `MapScreen.kt:148-174` — green hue for front, yellow for center, red for back. Placed pins are visible across taps; unplaced targets render nothing.

**Camera (v1)**:
- The map opens at a default world view; the user pans/zooms to the course themselves.
- v1 does NOT auto-center on GPS. This was originally planned but adds the coupling of injecting `LocationService` into the composable; we opted to defer it. Tracked under "Out of Scope (v1)".

**Why one map across all holes**: a single `GoogleMap` composable is kept across hole transitions; only marker state is re-keyed. Tile cache survives, so navigating 1 → 2 → 3 doesn't refetch tiles.

## Save & Error Handling

**Save flow** (executed when `onSave` is called from Review):

```kotlin
fun onSave() = viewModelScope.launch {
    val s = _uiState.value
    _uiState.update { it.copy(isSaving = true, saveError = null) }

    val course = Course(
        name = s.name.trim(),
        city = s.city.trim().ifBlank { null },
        state = s.state.trim().ifBlank { null },
    )
    val holes = (1..s.holeCount).map { n ->
        val g = s.greens[n - 1]
        HoleInfo(
            courseId = 0, holeNumber = n, par = s.pars[n - 1],
            greenFrontLat = g.frontLat,  greenFrontLng = g.frontLng,
            greenCenterLat = g.centerLat, greenCenterLng = g.centerLng,
            greenBackLat = g.backLat,    greenBackLng = g.backLng,
        )
    }
    val tees = s.tees.map { Tee(courseId = 0, name = it.name, color = it.color,
                                rating = it.rating, slope = it.slope) }

    runCatching { repo.createCourse(course, holes, tees) }
        .onSuccess { _events.emit(CreateCourseEvent.Saved) }
        .onFailure { e ->
            _uiState.update { it.copy(isSaving = false,
                                      saveError = e.message ?: "Save failed") }
        }
}
```

### Error surfaces

| Error                     | Where caught                                                              | UX                                                                                  |
|---------------------------|---------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| Blank name                | `onNext` from Name step                                                   | Red supporting text on TextField; Next disabled                                     |
| Duplicate course name     | `onNext` from Name step (`repo.getCourseByName`)                          | Red supporting text "A course with this name already exists"; Next disabled         |
| Invalid rating/slope      | `onNext` from Tees step / inline tee dialog Save                          | Error text in tee dialog (matches existing `TeeFormState.error` pattern)            |
| Save (DB) failure         | `onSave` `runCatching`                                                    | Snackbar on Review step; user stays, can retry Save                                 |
| Location unavailable      | First entry to Greens step                                                | Snackbar; map opens at default view, user pans manually                             |
| Process death mid-wizard  | n/a — not handled in v1                                                   | Lose draft, lands on CourseManagement                                               |

### Discard confirmation

Hardware back / top-bar back from Name step → `showDiscardDialog = true`. The AlertDialog uses the same pattern as `CourseManagementScreen.kt:112-128`:

- "Discard course? You'll lose your progress."
- Confirm → `popBackStack()`
- Dismiss → close dialog, stay on Name

From any step beyond Name, back goes one position rather than discarding outright. The dialog only appears when the user backs out of step 1.

### Unit tests — `CreateCourseViewModelTest`

Use a fake `CourseRepository` (preferred over mocks — assert real domain objects passed in). Coroutine test scope via `runTest`.

State machine:
- `onNext` from `Name` with blank name → stays on `Name`, `nameError = "blank"`.
- `onNext` from `Name` with duplicate (fake repo returns existing) → stays, `nameError = "duplicate"`.
- `onNext` from `Name` valid + unique → advances to `HoleCount`.
- `setHoleCount(9)` then advance to Pars → `pars` and `greens` both have size 9.
- `setHoleCount(18)` after previously setting 9 → both lists have size 18 (padded with defaults).
- `onNext` from `Greens` with incomplete `currentGreenHole` → stays on Greens, `currentGreenHole` unchanged.
- `onNext` from `Greens` with complete hole and `currentGreenHole < holeCount` → still Greens, hole incremented.
- `onNext` from `Greens` with complete hole and `currentGreenHole == holeCount` → advances to `Tees`.
- `onBack` from `Greens` with `currentGreenHole > 1` → decrements hole, step unchanged.
- `onBack` from `Greens` with `currentGreenHole == 1` → returns to `Pars`.
- `placeGreen(Front, latLng)` on `currentGreenHole = 2` → updates `greens[1].frontLat/Lng`, auto-advances `selectedGreenTarget` to `Center`.
- `placeGreen` on a target whose chip was clicked to clear → places at new latLng, advances target.
- `onNext` from `Tees` with zero tees → stays, error surfaced.
- `onSave` happy path → emits `Saved` event; fake repo's `createCourse` received correctly assembled `Course`, `List<HoleInfo>` of size `holeCount`, `List<Tee>` matching drafts.
- `onSave` failure (fake throws) → `saveError` set, `isSaving = false`, no event emitted.
- `onBack` from Name → `showDiscardDialog = true`.

### Repository tests — `CourseRepositoryImplTest`

In-memory Room DB:
- `createCourse_happyPath` — call with a course + 18 holes + 2 tees; assert all rows present with correct foreign keys.
- `createCourse_atomicity` — inject a DAO wrapper that throws on the tees insert; assert no `courses`, `hole_info`, or `tees` rows exist afterward (transaction rollback).

### No Compose UI tests in v1

The map step is hard to test in Compose UI tests (requires Google Play Services and a real tile renderer). The other steps are thin wrappers over state already covered by ViewModel tests.

## Out of Scope (v1)

These were considered and deferred:

- Editing green coordinates on existing courses (manual or OSM-imported).
- GPS capture by walking to the green (only tap-on-map is supported).
- Arbitrary hole counts (only 9 or 18).
- Geocoder lookup for course location (map centers on current GPS only).
- Process-death state preservation.
- Sharing manually-created courses with other users / uploading to OSM.
- Auto-centering the green-placement map on the user's current GPS location (was originally in scope; deferred for v1).
